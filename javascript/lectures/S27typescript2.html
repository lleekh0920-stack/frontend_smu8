Chapter 01
타입스크립트의 등장 배경

1.1. 자바스크립트의 장점과 한계

자바스크립트는 웹 브라우저에서 동작하는 언어로 시작했지만, 현재는 프론트엔드, 백엔드, 모바일 앱, 데스크톱 앱 개발 등 다양한 영역에서 사용된다. 자바스크립트의 가장 큰 장점 중 하나는 유연성이다. 변수에 문자열을 넣었다가 숫자를 넣을 수도 있고, 객체에 새로운 속성을 동적으로 추가할 수도 있다.

하지만 이러한 유연성은 프로젝트 규모가 커질수록 단점이 되기도 한다. 코드가 짧을 때는 문제가 없지만, 파일 수가 많아지고 여러 개발자가 함께 작업하면 “이 변수에는 어떤 값이 들어오는가”, “이 함수는 어떤 형태의 객체를 받아야 하는가”, “이 속성은 항상 존재하는가”를 파악하기 어려워진다. 순수 자바스크립트는 동적 타입 언어이므로, 많은 오류가 코드를 실행한 뒤에야 드러난다.

function printUserName(user) {
  console.log(user.name.toUpperCase());
}

printUserName({ name: "Kim" });
printUserName(null); // 실행 중 오류 발생 가능

위 코드에서 첫 번째 호출은 정상적으로 실행된다. 하지만 두 번째 호출에서는 null에 name 속성이 없기 때문에 실행 중 오류가 발생한다. 문제는 이 오류가 코드 작성 단계에서 잘 드러나지 않을 수 있다는 점이다.

1.2. 타입스크립트의 등장 이유

타입스크립트는 이러한 문제를 줄이기 위해 등장했다. 타입스크립트는 자바스크립트 문법을 기반으로 하면서, 변수·함수 매개변수·객체·반환값 등에 타입을 명시할 수 있게 한다. 또한 타입을 직접 쓰지 않아도 값에 따라 타입을 추론하는 기능을 제공한다.

function printUserName(user: { name: string }) {
  console.log(user.name.toUpperCase());
}

printUserName({ name: "Kim" });
// printUserName(null); // strict 모드에서는 오류로 검사됨
핵심 요약

타입스크립트는 자바스크립트를 완전히 대체하는 별개의 실행 언어라기보다, 자바스크립트에 정적 타입 검사 기능을 추가하여 코드를 더 안전하고 예측 가능하게 작성하도록 돕는 언어이다.

Chapter 02
왜 타입스크립트를 써야 하는가

2.1. 오류를 실행 전에 발견할 수 있다

타입스크립트의 가장 중요한 장점은 코드 실행 전에 타입 관련 오류를 발견할 수 있다는 점이다. 문자열을 받아야 하는 함수에 숫자를 전달하거나, 존재하지 않는 객체 속성에 접근하는 코드를 작성하면 타입스크립트가 이를 알려준다.

function greet(name: string): string {
  return `안녕하세요, ${name}님`;
}

greet("민수");
// greet(10); // 오류: number는 string 매개변수에 전달할 수 없음

2.2. 코드의 의도가 명확해진다

타입은 코드의 설명서 역할을 한다. 함수 이름만 보고는 어떤 값을 넣어야 하는지 알기 어렵지만, 타입이 있으면 함수의 사용 방법을 바로 파악할 수 있다.

function calculateDiscount(price: number, rate: number): number {
  return price * (1 - rate);
}

위 함수는 price와 rate가 숫자이고, 반환값도 숫자라는 사실을 명확하게 보여준다.

2.3. 자동완성과 리팩터링이 편리해진다

타입스크립트는 코드 편집기와 잘 연동된다. 타입 정보를 바탕으로 자동완성, 속성 추천, 함수 인자 안내, 잘못된 코드 표시 등을 제공한다.

type User = {
  id: number;
  name: string;
  email: string;
};

const user: User = {
  id: 1,
  name: "Kim",
  email: "kim@example.com"
};

console.log(user.email);

2.4. 큰 프로젝트에서 유지보수가 쉬워진다

프로젝트가 커질수록 타입의 중요성은 커진다. 여러 사람이 함께 개발할 때 함수의 매개변수나 객체 구조가 명확하지 않으면 코드를 이해하는 데 시간이 오래 걸린다. 타입스크립트는 데이터 구조를 명시적으로 표현하므로, 다른 개발자가 코드를 읽고 수정하기 쉬워진다.

주의

타입스크립트는 모든 버그를 막아 주는 도구가 아니다. 타입스크립트는 주로 타입 구조와 관련된 오류를 줄여 주며, 실제 비즈니스 로직 오류나 런타임 데이터 검증은 별도의 테스트와 검증 코드가 필요하다.

Chapter 03
타입스크립트 사용 방법

3.1. 프로젝트 준비

타입스크립트를 사용하는 전통적인 방법은 Node.js 프로젝트에서 타입스크립트 패키지를 설치하고, tsc 명령어로 타입스크립트 코드를 자바스크립트 코드로 변환하는 것이다.

npm init -y
npm install -D typescript
npx tsc --init

npx tsc --init 명령어를 실행하면 tsconfig.json 파일이 생성된다. 이 파일은 타입스크립트 컴파일 설정을 관리한다.

3.2. 기본 폴더 구조

typescript-basic/
├─ src/
│  └─ index.ts
├─ package.json
└─ tsconfig.json

src/index.ts 파일에 타입스크립트 코드를 작성하고, 컴파일하면 자바스크립트 파일이 생성된다.

3.3. tsconfig.json 기본 설정

{
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "rootDir": "src",
    "outDir": "dist",
    "strict": true
  },
  "include": ["src"]
}

target은 변환될 자바스크립트 버전을 의미하고, module은 모듈 방식을 의미한다. rootDir은 원본 타입스크립트 파일 위치, outDir은 변환된 자바스크립트 파일이 저장될 위치이다. strict는 엄격한 타입 검사를 켜는 옵션이므로 학습 단계에서도 켜 두는 것이 좋다.

3.4. 전통적인 컴파일과 실행

const message: string = "TypeScript 시작";
console.log(message);
npx tsc
node dist/index.js

전통적인 방식에서는 타입스크립트 코드를 먼저 자바스크립트 파일로 변환하고, 변환된 자바스크립트 파일을 실행한다.

Chapter 04
타입스크립트 파일을 바로 실행할 수 있는 이유

4.1. “변환 없이 실행”이라는 표현의 정확한 의미

최근에는 node example.ts, deno run main.ts, bun run index.ts처럼 타입스크립트 파일을 직접 실행하는 방식이 자주 사용된다. 이때 “자바스크립트 변환 없이 실행한다”는 표현은 보통 개발자가 npx tsc 같은 별도 컴파일 명령을 직접 실행하지 않아도 된다는 의미이다.

다만 실제 원리는 실행 환경마다 다르다. 타입스크립트의 타입 문법은 런타임에서 실행되는 값이 아니므로, 실행 환경은 타입 문법을 제거하거나 내부적으로 자바스크립트로 즉시 변환한 뒤 실행한다. 즉, 사용자가 직접 dist/index.js를 만들지 않을 뿐, 실행 가능한 자바스크립트 형태로 해석되는 과정은 여전히 필요하다.

핵심 정리

타입스크립트의 타입 정보는 런타임에 실행되는 코드가 아니다. 따라서 직접 실행 환경은 타입 주석을 제거하거나, 내부적으로 빠르게 변환하거나, 실행 가능한 자바스크립트 부분만 해석하는 방식으로 동작한다.

4.2. Node.js에서 직접 실행

Node.js의 최신 버전에서는 조건을 만족하는 타입스크립트 파일을 플래그 없이 직접 실행할 수 있다. 예를 들어 Node.js v22.18.0 이상에서는 제거 가능한 타입스크립트 문법만 포함된 파일을 다음과 같이 실행할 수 있다.

node example.ts

여기서 중요한 표현은 “제거 가능한 타입스크립트 문법”이다. 예를 들어 변수 타입 주석, 함수 매개변수 타입, 인터페이스, 타입 별칭처럼 실행 시점에 필요 없는 문법은 제거해도 자바스크립트 실행 의미가 유지된다.

function add(a: number, b: number): number {
  return a + b;
}

console.log(add(10, 20));

위 코드에서 : number 같은 타입 주석은 실행 시점에는 필요하지 않다. Node.js는 이러한 타입 문법을 제거한 뒤 실행할 수 있다. 반면 enum, 일부 namespace처럼 단순히 지우는 것만으로는 자바스크립트 코드가 되지 않는 문법은 별도의 변환이 필요할 수 있다.

node --experimental-transform-types another-example.ts

Node.js의 타입스크립트 실행 기능은 tsconfig.json을 실행 시점에 읽어 프로젝트 전체를 변환하는 방식이 아니다. 따라서 경로 별칭, 오래된 자바스크립트 버전으로의 다운레벨 변환, TSX 실행 등 일반적인 빌드 도구가 제공하는 기능과는 차이가 있다.

4.3. Deno에서 직접 실행

Deno는 타입스크립트를 일급 언어로 지원한다. 별도의 타입스크립트 설치 없이 .ts 파일을 실행하거나 가져올 수 있으며, 내부적으로 타입스크립트 코드를 자바스크립트로 컴파일해 실행한다.

deno run main.ts

다만 Deno는 실행 속도를 위해 기본적으로 deno run에서 타입 검사를 생략하고 코드를 실행한다. 실행 전에 타입 검사까지 수행하려면 다음과 같이 --check 옵션을 사용할 수 있다.

deno run --check main.ts

4.4. Bun에서 직접 실행

Bun도 타입스크립트와 JSX를 기본적으로 지원한다. Bun은 파일을 실행하기 전에 빠른 네이티브 트랜스파일러로 즉시 변환한 뒤 실행한다.

bun run index.ts
bun index.ts

4.5. 직접 실행과 타입 검사는 같은 것이 아니다

타입스크립트 파일을 바로 실행할 수 있다는 말은 항상 타입 검사가 먼저 수행된다는 뜻이 아니다. 많은 런타임은 실행 속도를 위해 타입 문법을 제거하거나 변환한 뒤 실행하고, 타입 검사는 별도의 명령으로 수행하게 한다.

npx tsc --noEmit

--noEmit은 자바스크립트 파일을 생성하지 않고 타입 검사만 수행할 때 사용할 수 있다. 최신 실행 환경에서 .ts 파일을 바로 실행하더라도, 실제 프로젝트에서는 tsc --noEmit 또는 빌드 도구의 타입 검사 과정을 함께 사용하는 것이 좋다.

실행 환경 실행 방식 주의점
Node.js 제거 가능한 타입 문법을 제거하고 실행 tsconfig.json 기반 변환이 아니며, 모든 타입스크립트 문법을 지원하는 것은 아님
Deno 내장 TypeScript 컴파일러로 내부 변환 후 실행 deno run은 기본적으로 타입 검사를 건너뛸 수 있으며, 필요 시 --check 사용
Bun 빠른 네이티브 트랜스파일러로 즉시 변환 후 실행 실행과 타입 검사를 구분해서 이해해야 함

Chapter 05
주요 문법

5.1. 기본 타입

타입스크립트에서 자주 사용하는 기본 타입은 string, number, boolean, null, undefined 등이다.

let username: string = "Kim";
let age: number = 20;
let isStudent: boolean = true;

5.2. 타입 추론

타입스크립트는 변수에 값을 대입하면 그 값을 기준으로 타입을 추론할 수 있다. 따라서 모든 변수에 타입을 직접 적을 필요는 없다.

let language = "TypeScript"; // string으로 추론
let count = 10;              // number로 추론

5.3. 배열 타입

let numbers: number[] = [1, 2, 3];
let names: Array<string> = ["Kim", "Lee", "Park"];

5.4. 튜플 타입

튜플은 배열처럼 생겼지만, 각 위치의 타입과 개수가 정해진 자료형이다.

let userInfo: [number, string] = [1, "Kim"];
// userInfo = ["Kim", 1]; // 오류

5.5. 객체 타입

let user: { id: number; name: string; email: string } = {
  id: 1,
  name: "Kim",
  email: "kim@example.com"
};

5.6. 선택적 속성

객체의 어떤 속성이 있을 수도 있고 없을 수도 있다면 속성 이름 뒤에 ?를 붙인다.

type User = {

Chapter 06
컴파일 결과물: .js, .map, .d.ts

6.1. 컴파일 결과물을 생성하는 설정

타입스크립트를 컴파일하면 기본적으로 자바스크립트 파일이 생성된다. 설정에 따라 .js.map, .d.ts, .d.ts.map 파일도 함께 생성할 수 있다.

{
  "compilerOptions": {
    "target": "ES2020",
    "module": "NodeNext",
    "rootDir": "src",
    "outDir": "dist",
    "strict": true,
    "sourceMap": true,
    "declaration": true,
    "declarationMap": true
  },
  "include": ["src"]
}

예를 들어 다음 타입스크립트 파일이 있다고 가정한다.

// src/math.ts
export function add(a: number, b: number): number {
  return a + b;
}

위 파일을 컴파일하면 설정에 따라 다음과 같은 파일들이 생성될 수 있다.

dist/
├─ math.js
├─ math.js.map
├─ math.d.ts
└─ math.d.ts.map

6.2. .js 파일

.js 파일은 실제로 실행되는 자바스크립트 파일이다. 타입스크립트의 타입 주석은 런타임에 필요하지 않으므로 제거된다.

// dist/math.js
export function add(a, b) {
  return a + b;
}

브라우저, Node.js 등 자바스크립트 실행 환경은 최종적으로 이와 같은 자바스크립트 코드를 실행한다.

6.3. .js.map 파일

.js.map 파일은 소스맵 파일이다. 소스맵은 컴파일된 자바스크립트 코드와 원본 타입스크립트 코드의 위치를 연결해 주는 지도 역할을 한다. 타입스크립트 공식 문서에 따르면 sourceMap 옵션은 디버거와 도구가 실제로는 변환된 자바스크립트를 다루더라도 원본 타입스크립트 소스 코드를 표시할 수 있도록 .js.map 파일을 생성한다.

// dist/math.js의 마지막 부분 예시
//# sourceMappingURL=math.js.map

예를 들어 브라우저 개발자 도구나 VS Code 디버거에서 오류가 발생했을 때, 실제 실행 파일은 dist/math.js이지만 개발자는 src/math.ts의 원래 줄 번호를 기준으로 문제를 확인할 수 있다.

추가 설명

.js.map 파일은 실행에 필수인 파일은 아니다. 그러나 디버깅을 편하게 해 주는 보조 파일이다. 배포 환경에서는 소스 코드 노출 가능성을 고려하여 소스맵 공개 여부를 결정해야 한다.

6.4. .d.ts 파일

.d.ts 파일은 선언 파일이다. 선언 파일은 구현 코드를 담지 않고 타입 정보만 담는다. 타입스크립트 공식 문서에서는 .d.ts 파일이 모듈의 외부 API를 설명하는 타입 정의 파일이며, 도구가 인텔리센스와 정확한 타입 정보를 제공하는 데 사용된다고 설명한다.

// dist/math.d.ts
export declare function add(a: number, b: number): number;

위 선언 파일에는 add 함수의 구현인 return a + b가 없다. 대신 “이 모듈은 숫자 두 개를 받아 숫자를 반환하는 add 함수를 제공한다”는 타입 정보만 담고 있다.

.d.ts 파일은 특히 라이브러리를 만들 때 중요하다. 다른 개발자가 이 라이브러리를 타입스크립트 프로젝트에서 사용할 때, 타입스크립트와 편집기는 .d.ts 파일을 보고 함수의 매개변수, 반환값, 객체 구조를 이해한다.

6.5. .d.ts.map 파일

.d.ts.map 파일은 선언 파일용 소스맵이다. .d.ts 파일의 타입 선언이 원본 .ts 파일의 어느 위치에서 왔는지 연결해 준다. 이 파일이 있으면 편집기에서 “정의로 이동” 같은 기능을 사용할 때 생성된 선언 파일이 아니라 원본 타입스크립트 파일로 이동하는 데 도움이 된다.

6.6. 생성 파일 비교

파일 생성 옵션 역할 실행 필요 여부
.js 기본 컴파일 결과 실제로 실행되는 자바스크립트 코드 필요
.js.map sourceMap: true 컴파일된 JS와 원본 TS의 위치를 연결하는 디버깅 지도 실행 자체에는 불필요
.d.ts declaration: true 외부에서 사용할 타입 정보 제공 실행 자체에는 불필요
.d.ts.map declarationMap: true 선언 파일과 원본 TS 파일의 위치 연결 실행 자체에는 불필요

6.7. 어떤 경우에 생성하면 좋은가

애플리케이션 개발에서는 디버깅을 위해 sourceMap을 켜는 경우가 많다. 다만 운영 배포 환경에서는 소스맵 공개 여부를 신중하게 결정해야 한다. 라이브러리 개발에서는 사용자에게 타입 정보를 제공해야 하므로 declaration을 켜는 것이 일반적이다. 프로젝트 참조나 모노레포 환경에서는 declarationMap을 함께 사용하면 편집기 이동 기능이 편리해진다.

간단한 비유

.js가 실제 제품이라면, .js.map은 제품의 어느 부분이 설계도 어느 위치에서 왔는지 알려 주는 지도이고, .d.ts는 제품을 사용하는 사람을 위한 사용 설명서에 가깝다.

학습 정리

타입스크립트는 자바스크립트의 유연함을 유지하면서 타입 안정성을 추가한 언어이다. 자바스크립트 코드가 커질수록 변수와 객체의 구조를 예측하기 어려워지는데, 타입스크립트는 타입을 통해 이러한 문제를 줄인다. 타입스크립트의 핵심은 타입을 많이 쓰는 것이 아니라, 코드의 의도를 명확하게 표현하고 잘못된 사용을 미리 막는 것이다.

최근에는 Node.js, Deno, Bun 같은 실행 환경에서 타입스크립트 파일을 직접 실행할 수 있다. 그러나 이 말은 타입스크립트 타입 문법이 런타임에 그대로 실행된다는 뜻이 아니다. 실행 환경은 타입 문법을 제거하거나 내부적으로 빠르게 변환하여 실행한다. 따라서 실무에서는 “직접 실행”, “타입 검사”, “배포용 빌드”를 구분해서 이해해야 한다.

컴파일 결과물 중 .js는 실제 실행 파일, .js.map은 디버깅을 위한 소스맵, .d.ts는 외부 사용자를 위한 타입 선언 파일, .d.ts.map은 선언 파일과 원본 타입스크립트 파일을 연결하는 선언 소스맵이다.

}

문제 8 정답

타입스크립트의 타입 주석과 인터페이스 같은 문법은 런타임에서 실행되는 값이 아니므로 제거할 수 있다. 최신 실행 환경은 타입 문법을 제거하거나 내부적으로 자바스크립트로 즉시 변환한 뒤 실행한다. 따라서 개발자가 직접 tsc로 .js 파일을 만들지 않아도 .ts 파일을 실행할 수 있다. 다만 실행과 타입 검사는 같은 것이 아니므로, 필요하면 별도의 타입 검사 명령을 사용해야 한다.

문제 9 정답

.js.map 파일은 컴파일된 자바스크립트 코드와 원본 타입스크립트 코드의 위치를 연결해 주는 소스맵 파일이다. 디버깅할 때 실행 파일은 .js이지만, 개발자는 원본 .ts 파일의 줄 번호를 기준으로 오류를 확인할 수 있다.

문제 10 정답

.d.ts 파일은 구현 코드를 담지 않고 타입 정보만 담는 선언 파일이다. 라이브러리나 모듈을 사용하는 쪽에서 함수 매개변수, 반환값, 객체 구조를 알 수 있도록 도와준다.

문제 11 정답

.d.ts.map 파일은 선언 파일의 타입 선언이 원본 .ts 파일의 어느 위치에서 왔는지 연결해 준다. 편집기에서 “정의로 이동” 기능을 사용할 때 생성된 선언 파일이 아니라 원본 타입스크립트 파일로 이동할 수 있어 모노레포나 라이브러리 개발에서 유용하다.

문제 12 정답

제네릭은 함수, 클래스, 타입을 특정 타입 하나에 고정하지 않고 여러 타입에서 재사용하기 위해 사용한다.

function getFirst<T>(items: T[]): T {
  return items[0];
}

const firstNumber = getFirst([1, 2, 3]);
const firstName = getFirst(["Kim", "Lee"]);
