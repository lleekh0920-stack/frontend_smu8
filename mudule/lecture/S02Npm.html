1. 패키지 관리자란?

Node.js 프로젝트에서는 모든 기능을 직접 작성하지 않고, 다른 개발자가 만들어 둔 라이브러리나 도구를 가져와 사용한다. 이때 외부 코드를 검색하고, 다운로드하고, 프로젝트에 연결하고, 버전을 관리해 주는 도구를 패키지 관리자라고 한다. 대표적인 Node.js 패키지 관리 도구에는 npm, Yarn, pnpm이 있다.

패키지 관리자는 단순히 파일을 내려받는 도구가 아니다. 프로젝트가 어떤 외부 패키지에 의존하는지 기록하고, 정확히 어떤 버전이 설치되었는지 고정하며, 다른 개발자나 배포 서버에서도 같은 환경을 재현할 수 있도록 돕는다.

핵심은 “필요한 패키지를 내려받는다”가 아니라 “프로젝트가 사용하는 외부 코드의 목록과 버전을 일관되게 관리한다”이다.

2. Node.js 패키지 관리의 기본 구조

2.1. package.json

package.json은 Node.js 프로젝트의 설정 파일이다. 프로젝트 이름, 버전, 실행 명령어, 사용하는 패키지 목록 등이 이 파일에 기록된다. 특히 프로젝트가 의존하는 패키지를 나열하고, 사용할 수 있는 패키지 버전을 지정하며, 프로젝트 실행 및 빌드 환경을 재현 가능하게 만드는 역할을 한다.

{
  "name": "node-package-demo",
  "version": "1.0.0",
  "scripts": {
    "start": "node index.js",
    "dev": "nodemon index.js"
  },
  "dependencies": {
    "lodash": "^4.17.21"
  },
  "devDependencies": {
    "nodemon": "^3.1.0"
  }
}

dependencies에는 실제 애플리케이션 실행에 필요한 패키지를 기록한다. 예를 들어 웹 서버를 만드는 express, 날짜를 다루는 dayjs, 유틸리티 함수가 많은 lodash 같은 패키지가 여기에 들어갈 수 있다.

devDependencies에는 개발 중에만 필요한 도구를 기록한다. 예를 들어 코드 변경을 감지해 서버를 다시 실행하는 nodemon, 테스트 도구, 빌드 도구 등이 여기에 들어갈 수 있다.

2.2. node_modules

node_modules는 실제 패키지 파일들이 설치되는 폴더이다. npm install을 실행하면 package.json에 기록된 패키지들이 로컬 node_modules 폴더에 설치된다.

node_modules 폴더는 크기가 매우 커질 수 있고, 설치 명령어로 다시 만들 수 있다. 일반적으로 Git 같은 버전 관리 도구에는 node_modules를 올리지 않고, package.json과 lockfile을 올린다.

2.3. lockfile

lockfile은 실제로 설치된 패키지의 정확한 버전을 고정하는 파일이다. npm은 package-lock.json, Yarn은 yarn.lock, pnpm은 pnpm-lock.yaml을 사용한다.

package.json이 “이 범위의 버전을 사용할 수 있다”를 의미한다면, lockfile은 “이번 프로젝트에서는 실제로 이 버전을 설치했다”를 의미한다. 예를 들어 package.json에는 ^4.17.0처럼 범위가 기록될 수 있지만, lockfile에는 실제 설치된 4.17.21 같은 정확한 버전이 기록된다.

package.json은 의존성의 선언서이고, lockfile은 실제 설치 결과의 고정 기록이다.

2.4. 패키지 설치 과정

패키지 설치 과정은 보통 네 단계로 이해할 수 있다. 첫째, 패키지 관리자가 package.json과 lockfile을 읽는다. 둘째, 사용할 패키지 버전을 결정한다. 셋째, npm registry 같은 저장소나 캐시에서 패키지를 가져온다. 넷째, 프로젝트에서 사용할 수 있도록 node_modules 또는 각 패키지 관리자의 방식에 맞게 연결한다.

이 과정에서 버전 범위가 중요하다. 시맨틱 버전은 보통 주 버전, 부 버전, 패치 버전의 개념으로 관리된다. 일반적으로 호환되는 버그 수정은 패치 버전, 호환되는 기능 추가는 부 버전, 호환성이 깨지는 변경은 주 버전을 올리는 방식으로 이해할 수 있다.

3. 주요 패키지 관리자

Yarn JavaScript 프로젝트의 의존성 설치, 업데이트, 제거 등을 관리하는 패키지 관리자 yarn.lock Yarn 기반 프로젝트, 워크스페이스, Zero-Installs 등

Corepack 패키지 관리자 버전을 프로젝트별로 맞추는 보조 도구 직접 lockfile을 만들지는 않음 프로젝트별 패키지 관리자 버전 고정

npm은 Node.js를 시작할 때 가장 자주 접하게 되는 기본 패키지 관리자이다. Yarn과 pnpm은 npm의 대안으로 사용되며, 설치 속도, 의존성 관리 방식, 모노레포 지원 등에서 각각의 특징을 가진다.

4. Node.js와 npm 설치 확인

Node.js 패키지를 설치하려면 먼저 Node.js와 npm CLI가 필요하다. 설치 여부와 버전은 다음 명령어로 확인할 수 있다.

node -v
npm -v

정상적으로 설치되어 있다면 다음과 비슷한 결과가 출력된다.

v22.0.0
10.0.0

버전 숫자는 설치 시점과 환경에 따라 다르다. 중요한 것은 두 명령어가 모두 실행되어야 한다는 점이다.

Yarn이나 pnpm을 사용할 때는 Corepack을 사용할 수 있다. Corepack이 준비되어 있다면 다음 명령어로 활성화한다.

corepack enable

프로젝트에서 Yarn을 사용하려면 다음과 같이 설정할 수 있다.

corepack use yarn@stable

프로젝트에서 pnpm을 사용하려면 다음과 같이 설정할 수 있다.

corepack use pnpm@latest

Corepack은 Yarn과 pnpm 실행 파일을 경로에 연결하고, 프로젝트가 사용할 패키지 관리자 정보를 package.json에 기록하는 데 활용할 수 있다.

5. npm 사용법

5.1. 프로젝트 생성

먼저 새 폴더를 만들고 프로젝트를 초기화한다.

mkdir node-package-demo
cd node-package-demo
npm init -y

npm init -y는 기본값으로 package.json을 생성하는 명령어이다.

5.2. 패키지 설치

예를 들어 lodash 패키지를 설치하려면 다음과 같이 입력한다.

npm install lodash

이 명령어를 실행하면 다음과 같은 변화가 생긴다.

node-package-demo/
├─ node_modules/
├─ package.json
└─ package-lock.json

lodash는 node_modules 안에 설치되고, package.json의 dependencies에 기록되며, package-lock.json에는 정확한 설치 버전과 의존성 트리가 기록된다.

간단히 사용해 보면 다음과 같다.

const _ = require("lodash");

const result = _.camelCase("hello package manager");
console.log(result);

파일 이름을 index.js로 저장한 뒤 실행한다.

node index.js

실행 결과는 다음과 같다.

helloPackageManager

5.3. 개발용 패키지 설치

개발 중에만 필요한 패키지는 -D 옵션으로 설치한다.

npm install -D nodemon

이 명령어는 nodemon을 devDependencies에 기록한다.

{
  "devDependencies": {
    "nodemon": "^3.1.0"
  }
}

이제 package.json의 scripts에 실행 명령을 추가할 수 있다.

{
  "scripts": {
    "start": "node index.js",
    "dev": "nodemon index.js"
  }
}

실행은 다음과 같이 한다.

npm run dev

npm run은 package.json의 scripts 객체에 정의된 명령어를 실행한다.

5.4. 기존 프로젝트의 패키지 설치

다른 사람이 만든 프로젝트를 내려받았을 때 node_modules가 없더라도 package.json과 lockfile이 있으면 패키지를 다시 설치할 수 있다.

npm install

이 명령어는 현재 프로젝트의 의존성을 설치한다.

배포 서버나 CI 환경처럼 항상 같은 설치 결과가 필요할 때는 다음 명령어를 사용한다.

npm ci

npm ci는 자동화 환경에서 깨끗한 설치를 하기 위한 명령어이다. 기존 node_modules가 있으면 제거하고, package-lock.json과 package.json이 맞지 않으면 lockfile을 수정하지 않고 오류로 종료한다.

5.5. 패키지 제거와 업데이트

패키지를 제거하려면 다음과 같이 입력한다.

npm uninstall lodash

패키지를 업데이트하려면 다음과 같이 입력한다.

npm update

특정 패키지만 업데이트하려면 패키지 이름을 붙인다.

npm update lodash

5.6. 패키지 정보 확인

패키지를 설치하기 전에 registry 정보를 확인할 수 있다.

npm view express
npm view express version
npm view express dependencies

npm view는 registry에 등록된 패키지 정보를 출력하는 명령어이다. 패키지 버전, 의존성, 저장소 정보 등을 확인할 때 사용할 수 있다.

5.7. 일회성 실행

패키지를 프로젝트 의존성으로 저장하지 않고 한 번만 실행하고 싶을 때는 npx를 사용할 수 있다.

npx cowsay "hello"

npx는 로컬 또는 원격 npm 패키지의 명령어를 실행할 수 있다. 필요한 패키지가 로컬 프로젝트에 없으면 npm 캐시에 설치한 뒤 실행할 수 있다.

6. Yarn 사용법

Yarn 프로젝트를 시작할 때는 Corepack을 활성화한 뒤 Yarn을 설정한다.

corepack enable
corepack use yarn@stable

새 Yarn 프로젝트는 다음과 같이 초기화할 수 있다.

yarn init -2

패키지 설치는 다음과 같이 한다.

yarn add lodash

개발용 패키지는 다음과 같이 설치한다.

yarn add -D nodemon

yarn add는 의존성을 프로젝트에 추가하는 명령어이며, 기본적으로 일반 dependencies에 추가하고 -D 또는 --dev 옵션을 사용하면 devDependencies에 추가한다.

기존 Yarn 프로젝트의 의존성을 설치할 때는 다음 명령어를 사용한다.

yarn install

패키지를 제거하려면 다음과 같이 입력한다.

yarn remove lodash

패키지를 업데이트하려면 다음과 같이 입력한다.

yarn up lodash

7. pnpm 사용법

pnpm을 사용할 때도 Corepack으로 프로젝트에 설정할 수 있다.

corepack enable
corepack use pnpm@latest

패키지 설치는 다음과 같이 한다.

pnpm add lodash

개발용 패키지는 다음과 같이 설치한다.

pnpm add -D nodemon

pnpm add <pkg>는 패키지와 그 패키지가 의존하는 패키지들을 설치한다. 기본적으로 새 패키지는 일반 dependencies에 저장되며, -D 옵션을 사용하면 devDependencies에 저장된다.

기존 pnpm 프로젝트의 의존성을 설치할 때는 다음 명령어를 사용한다.

pnpm install

패키지를 제거하려면 다음과 같이 입력한다.

pnpm remove lodash

패키지를 업데이트하려면 다음과 같이 입력한다.

pnpm update

pnpm은 특히 여러 프로젝트를 다루는 환경에서 디스크 공간을 절약하는 장점이 있어 많이 사용된다.

9. 패키지를 내려받는 여러 방식

가장 일반적인 방식은 npm registry에 등록된 패키지 이름으로 설치하는 것이다.

npm install express

특정 버전을 설치할 수도 있다.

npm install express@4.18.2

특정 태그를 기준으로 설치할 수도 있다.

npm install express@latest

개발용 의존성으로 설치하려면 다음과 같이 한다.

npm install -D nodemon

전역 설치가 필요한 CLI 도구는 -g 옵션으로 설치할 수 있다.

npm install -g typescript

단, 프로젝트마다 필요한 도구는 전역 설치보다 로컬 개발 의존성으로 설치하고 scripts를 통해 실행하는 방식이 더 일관적이다. 전역 설치는 여러 프로젝트에서 공통으로 사용하는 명령줄 도구에 제한적으로 사용하는 것이 좋다.

npm은 패키지 이름뿐 아니라 로컬 폴더, tarball 파일, tarball URL, registry의 특정 버전이나 태그, Git 원격 저장소 등 다양한 형태의 패키지 지정 방식을 지원한다.

11. 패키지 관리 시 주의할 점

package.json과 lockfile은 프로젝트의 의존성 정보를 담고 있으므로 버전 관리에 포함한다. lockfile은 팀원, 배포, CI 환경에서 같은 의존성을 설치하도록 보장하는 용도로 사용된다.

node_modules는 직접 수정하지 않는다. 패키지 코드에 문제가 있더라도 node_modules 안의 파일을 직접 고치면 재설치 시 변경 내용이 사라진다. 필요한 경우 패키지 버전을 바꾸거나, 직접 만든 코드를 별도 파일로 작성하거나, 패치 도구를 사용한다.

하나의 프로젝트에서는 하나의 패키지 관리자를 일관되게 사용하는 것이 좋다. npm을 사용하면 package-lock.json, Yarn을 사용하면 yarn.lock, pnpm을 사용하면 pnpm-lock.yaml이 만들어진다. 여러 lockfile이 섞이면 실제 설치 결과가 달라질 수 있다.

배포나 CI 환경에서는 가능한 한 npm ci, yarn install --immutable, pnpm install --frozen-lockfile처럼 lockfile을 기준으로 설치 결과를 고정하는 명령을 사용한다.

Node.js 패키지 관리자 소개와 사용법 문제 정답

문제1. 외부 라이브러리와 도구를 검색, 다운로드, 설치, 업데이트, 제거하고 프로젝트의 의존성 버전을 일관되게 관리하기 위해 필요하다.

문제2. package.json은 프로젝트 이름, 버전, 실행 스크립트, 의존성 목록, 사용할 수 있는 패키지 버전 범위 등을 기록하는 Node.js 프로젝트 설정 파일이다.

문제3. dependencies는 애플리케이션 실행에 필요한 패키지를 기록하는 영역이고, devDependencies는 개발 중에만 필요한 테스트 도구, 빌드 도구, 실행 보조 도구 등을 기록하는 영역이다.

문제4. node_modules에는 실제로 다운로드된 패키지 파일과 그 패키지들이 다시 의존하는 하위 패키지들이 저장된다.

문제5. lockfile은 실제 설치된 패키지의 정확한 버전과 의존성 구조를 기록하여, 다른 개발자나 배포 환경에서도 같은 패키지 구성을 재현할 수 있게 한다.

문제6.

npm install lodash
npm install -D nodemon

문제7.

npm install

문제8. npm install은 package.json과 lockfile을 참고해 의존성을 설치하며 필요하면 lockfile을 갱신할 수 있다. npm ci는 lockfile을 기준으로 깨끗하게 설치하고, package.json과 lockfile이 맞지 않으면 오류로 종료하며 lockfile을 수정하지 않는다.

문제9.

yarn add lodash
yarn remove lodash
yarn up lodash

문제10.

pnpm add lodash
pnpm add -D nodemon
